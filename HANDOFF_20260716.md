## 現在の状況

このセッションで実施し、既にpush済みの修正(`js/legacy/part1.js`, `part8.js`, `server/server.js`):

1. **landuse=residentialの無条件上書きバグ修正**: `generateChunk()`の`buildable()`(part8.js)が、landuseタグがresidentialなら本物の建物の有無を一切見ずに一戸建て補完を許可していた。日本のOSMのlanduse=residentialは粗く、大きな商業ビルの敷地まで覆っていることが多い。`isInsideKnownRealBuilding(qx,qz,pad)`(part1.js、`knownBuildingGrid`から実フットプリントを判定)を新設し、`buildable()`の最優先チェック、および`buildable()`を通らない別経路(2番目の区画内グリッド充填ループ)の両方に組み込んだ。→ 低層住宅の密集は大きく改善したとユーザー確認済み。

2. **Overpass巨大クエリの信頼性問題(数ラウンドにわたる調査)**: 低層住宅は減ったが、今度は「大きい建物が建つべき場所が空き地のまま」という新症状が発生。実機診断とOSMへの直接問い合わせ(Overpass API)を繰り返した結果、根本原因は **Overpassの公開インスタンスが、エラーもHTTPステータス異常もremark欄の警告も一切出さずに、その時の負荷次第で「たまたま応答できた分だけ」を通常の200 OKとして返すことがある**、という非決定的な挙動だと判明。新川・八丁堀エリアで実測: 周辺タイル全て`loaded:true, fail:0`(=完全成功扱い)なのに、実際の建物712件中631件(89%)がゲーム内のどこにも存在していなかった。一度この「見せかけの成功」でタイルが`loadedOSMTiles`に入ると、ページをリロードしない限り二度と再取得されない。

   対策として最終的に:
   - `buildOSMBatchQuery`が`{query, timeout}`を返すようにし、クライアント側のabort待ち時間をOverpassに指定した`[timeout:N]`と揃えた(以前は不一致で正常なクエリを誤って中断していた)。
   - `data.remark`にtimeout/memory系の文言があれば失敗として再試行キューに戻す処理を追加(ただし後述の通りこれだけでは不十分と判明)。
   - `OSM_TILE_BATCH`(1リクエストにまとめるタイル数)を 6 → 3 → **最終的に1** まで縮小。1タイルずつ取得するのが最も確実(検証時は毎回1〜1.5秒で完全なデータが返ってきた)。
   - `server.js`の直接アクセスモード(プロキシ不調時のフォールバック)のペース配分に競合状態(2並列で同時に古いタイムスタンプを読んでしまう)があったのを、プレフィックスごとのPromiseチェーンで直列化して修正。

   関連メモリ: `project_isehara_game_procedural_infill_race.md`(v1〜v7まで経緯を全部記録済み)。

## 試して駄目だったこと

- 6タイルまとめクエリ → 密集地で504 Gateway Timeout。
- 失敗したタイルだけ即座に1枚に縮小する対策 → 60タイル規模のバックログでリクエスト数が急増し、サーバー側キュー・直接モード双方が詰まって429/502が連鎖する新たな不具合を誘発(その後、閾値を「2回失敗したら」に緩和したが、根本解決には至らず)。
- 3タイル・4タイルまとめ → 検証時はたまたま成功したが、密集地では毎回安全ではないと後で判明(非決定的な部分応答問題のため)。
- `remark`欄の検知 → Overpassが本当に何も警告を出さずに部分応答を返すケースがあり、原理的に検出できない場合があった。
- 「関連(relation/マルチポリゴン)建物が原因では」という仮説 → 実データ突き合わせで否定(欠落建物は全部普通の`way`タイプだった)。
- 「外部のレート制限が原因では」という仮説 → 2時間待っても症状不変だったため否定。
