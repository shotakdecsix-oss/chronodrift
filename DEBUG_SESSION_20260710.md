## 1. 修正①: 全ファイル末尾欠損の復旧(致命的バグ)

**症状**: git追跡ファイル10個すべてが末尾数十バイト〜数KB欠損していた。
- `index.html` … 末尾76行(animate後半・`animate()`呼び出し・resize・閉じタグ)が消失 → JS構文エラーでゲーム起動不能
- `package.json` … JSON途中で切断 → `npm start`不能
- `server/server.js`、`.gitignore`、各.md も同様

**原因(推定)**: Cowork/Claudeセッションのbashサンドボックスのマウント同期不具合。
bash側がファイルサイズを古い値でキャッシュし、その長さで切り詰めた内容を見る/書く
(SESSION_LOG.md記載の「bashスナップショット固定」現象と同根)。

**対応**: 全ファイルが「HEADと同一+末尾欠損のみ」と diff で確認できたため、
HEAD(`a3ef243`)の内容で全復元。バイナリ比較で完全一致を確認。失われた作業なし。

**教訓(重要・再発防止)**:
- **bash(サンドボックス)から `index.html` 等リポジトリのファイルへ書き込み禁止**。
  bashのstat/サイズは編集後に古いまま固定されることがあり、その状態で書くと切り詰め破損する。
- 編集は必ず Edit/Write ツール(ホスト側)で行う。bashは読み取りも信用しきれない
  (今セッションでも編集後のファイルが bash からは末尾欠損に見えた)。
- 構文チェックは「git HEADをベースに同一置換をNodeスクリプトで再適用した複製」を
  sandbox内に作って `node --check` する方式が確実(置換が一意にマッチしなければ
  即例外=ホスト側と同一内容の保証になる)。

## 2. 修正②: 標高ソースを国土地理院(GSI)タイルへ切替

**背景**: opentopodata は 1リクエスト/秒+100地点/リクエスト+**1日1000コール上限**。
サーバープロキシが1.1秒間隔に直列化するため、初回起動だけで約21コール≒25秒。
さらに「地形(NEAR)が届くまで道路・建物生成をゲートする」設計のため、
標高の遅さが全読み込みの律速だった。

**実装**(`index.html`):
- `fetchElevationsGSI(latlons)` を新設。GSI標高タイル
  `https://cyberjapandata.gsi.go.jp/xyz/dem_png/14/{x}/{y}.png`(DEM10B相当、約10mメッシュ)を
  タイル単位で並列取得(同時8)し、canvasでデコード。
  標高式: `x = R*2^16 + G*2^8 + B`、`x==2^23`は無効(海)、`x>2^23`は`x-2^24`、単位0.01m。
- タイルキャッシュ `_gsiTiles`(Map、上限120枚≒30MB、Promiseをキャッシュ)。
- 適用箇所: `loadElevations()`(初期地形)と `loadNearTerrain()`(NEAR=建物生成のゲート)。
  **遠景 `loadWideTerrain()` は従来のopentopodataのまま**(±14kmはz14タイル約200枚必要で重いので見送り。
  次にやるなら `demgm_png` z8 の検証から)。
- フォールバック: 日本範囲外(lat 20-46 / lon 122-154 外)の点が混じる、または
  ネットワーク失敗時は null を返し、従来のopentopodata経路にそのまま落ちる。
  海上(404/無効画素)は「データ無し」→呼び出し側で従来どおり0m/海底扱い。

**実測検証**(実ブラウザ): 東京駅3.4m/伊勢原駅25.2m/富士山頂3755.8m/大山1248.7m/
琵琶湖84.6m(湖面も有効値が返る)、海上404、5地点デコード込み236ms。CORS問題なし。

## 3. 修正③: 道路メッシュ生成のフレーム分割

**背景**: `addRoad` が呼ばれた瞬間に `makeRoadGeo`(1mごと地形サンプリング)+Mesh生成+
`scene.add` を同期実行。密集市街地のタイル(6枚バッチ)到着時に数千セグメントを
1フレームで生成し、東京都心で45秒超のフリーズを確認。

**実装**(`index.html`):
- `pendingRoadMeshes` キュー+`queueRoadMesh(r)`+`processRoadMeshQueue()`(animateから毎フレーム、
  **1フレーム6msの時間バジェット**で処理)を新設。
- `addRoad` はレコード登録(`addRoadRecord`)+キュー投入のみに変更。
  当たり判定(`isOnRoad`)・ミニマップ・踏切検出はレコードで動くので即時有効。
  メッシュは数フレーム〜数秒遅れて出る(仕様変更)。
- `rebuildRoadsInBounds`(NEAR更新時の±4km一括再構築)も同期→キュー経由に変更
  (`r._dirty = true` で「メッシュ生成済みでも作り直し」)。
- キュー処理時、`ROAD_UNLOAD_DIST`(2.5km)より遠いものは生成せず破棄
  (`unloadFarRoads` がどうせ即解放する。近づけばチャンク再生成/NEAR更新が再キューする)。
- 非現実モードの線路白帯(railWhite)は `rebuildRoadMesh` が本体と同時生成(従来から対応済み)。

## 4. 修正④: 格子状フォールバック道路の根治(Render IP問題)

**症状**: デプロイ版で道路が読み込めず格子状フォールバックマップになる。
SESSION_LOG記載の既知現象「OSMグリッドフォールバック」も同根の可能性大。

**切り分け結果**: Overpass API本体は健在(ユーザーのブラウザから直接200・1.1秒)だが、
**Renderプロキシ経由だと502連発**(≒OverpassがRenderの共有IPを拒否/制限)。

**実装**(`server/server.js` の INJECT スクリプト):
- fetch書き換えに `proxyDown` フラグを追加。プロキシ応答が **5xx またはネットワーク失敗**
  だったAPIプレフィックスは、以後そのセッション中 **ブラウザ→上流の直接fetch** に切替。
  上流(Overpass/opentopodata/Nominatim)は全てCORS対応なので直接叩ける。
  各プレイヤー自身のIPでレート制限枠を使うため、共有IPより通りやすい。
- プロキシ健在時は従来どおりプロキシ+ディスクキャッシュ(ローカルプレイで有効)。

## 5. 修正⑤: マップジャンプ後の読み込み高速化

**症状**: ジャンプ後、道路まで1〜2分、建物出揃いまでさらに1〜2分。

**律速の分析**: ジャンプ後は7×7=49タイルを「6枚まとめクエリ×3並列」で取得するが、
Overpassの同時実行枠は**1IPあたり2つ**、6枚まとめクエリは1本20〜40秒 → 合計1〜2分。
建物は所属タイル到着後に生成されるため後ろへずれる(建物生成自体は可変バジェットで速い)。

**実装**(`index.html` `fetchOSMTileBatch`):
- **現在地タイルが未確定の間はバッチサイズ1**(小クエリを最優先発行→足元の道路・建物が
  約10〜20秒で出る)。現在地確定後は従来の6枚まとめ。
- **50秒のクライアント側タイムアウト**を追加(`Promise.race`)。直接アクセスモードには
  サーバー側の45秒タイムアウトが効かず、ハングしたリクエストがワーカー枠を永久に塞いでいた
  (実測90秒以上のpendingを確認)。失敗扱い→既存のバックオフ・再試行に乗る。
- 見送り: Overpassミラー(kumi.systems)併用は検証時に応答不安定のため不採用。

## 6. 現在の読み込みパイプライン(改修後の全体像)

1. 起動: `fetchOSMData()`(Overpass、初期範囲)と `loadElevations()`(**GSI、数秒**)が並行
2. `loadOSM()` → 道路レコード登録(即)+メッシュはフレーム分割で出現
3. `loadWideTerrain`(opentopodata、背景・遠景のみ)/ `loadNearTerrain`(**GSI、数秒**)
4. タイル取得: ジャンプ直後は1枚小クエリ→以後6枚まとめ、近い順、3ワーカー、50sタイムアウト
5. 建物: タイル到着後 `pendingBuildings` に積み、NEAR準備済みなら可変バジェット(20〜80棟/フレーム)で生成
6. Render上でプロキシ5xx検出時は自動でブラウザ→上流直行に切替(セッション毎)

## 7. 既知の残課題・次の一手

- **遠景(WIDE)標高はまだopentopodata**(9バッチ×1.1秒+日次上限を消費)。
  `demgm_png` z8(全球版、単位が異なる可能性あり要検証)への切替候補。
- 全域(±5.6km)のタイル充填は依然1〜2分(Overpassの枠制限による背景処理)。
  根本策はOSM公式ベクタータイル等への移行(大改修)。
- GSIのDEM10Bに稀にある無効画素(陸上)は0m/海底扱いになる(実害は未確認)。
- 湖面はGSIで有効値が返ることを確認済み(琵琶湖84.6m)。
- Overpass直行モードでは `OSM_TILE_CONCURRENCY = 3` > 同時枠2なので、3本目は429になりがち
  (既存のバックオフで回復するが、2に下げる選択肢もある)。
