# セッションログ(伊勢原ファンタジー・エンジン開発履歴)

このファイルは `index.html`(3D探索アプリ本体)と `server/server.js` の実装作業について、
セッションをまたいで引き継ぐための技術ログ。新しいチャットで作業を再開する際は、まずこのファイルの
「現在の状態」を読めば、何が入っていて何が入っていないかがわかる。

デザイン構想(文明育成シム)側の引き継ぎは `HANDOVER.md`/`GAME_DESIGN.md` を参照。本書はエンジン
(地図・地形・建物などのレンダリング/生成ロジック)側の作業ログ専用。

---

## 現在アクティブな機能(このログの最新時点)

- **道路・地形**: 実座標OSM道路+実標高地形。高速道路は橋脚なしで空中に浮く形で表示(橋脚は挙動が
  不安定だったため視覚・物理とも撤去済み)。高速道路の路面は連続式の勾配計算(`motorwaySlopes`)で
  当たり判定を出しており、階段状のガタつきは解消済み。
- **道路・線路の距離アンロード**: `ROAD_UNLOAD_DIST=2500` を超えた記録はメッシュだけGPU解放し、
  再訪時に再構築。長時間探索でのクラッシュ対策。
- **建物の外観**: `building:colour`/`roof:colour`/`roof:shape`/`roof:material` タグがあれば反映。
  `building:height`/`building:levels` を優先し、無ければランダム。高さ・フットプリントが閾値
  (`APT_HEIGHT_M=10`, `APT_FOOTPRINT_M2=200`)を超えると一戸建てタグでもマンション扱いに昇格。
- **学校・病院・役場・神社仏閣**は最低限の高さを底上げ(`applyLandmarkMinHeight`)してランドマーク
  として埋もれないようにしてある。マンション・工場は最低サイズを底上げ(`applySizeFloor`)。
  工場には煙突を追加。
- **タグ不明な一般建物(現実モード)**は幻想紫(`DEFAULT_WALLS`)ではなく、中立なグレー・ベージュ系
  (`DEFAULT_WALLS_REAL`)を使う。
- **住宅密集地の判定(`denseArea`)**: 細街路が格子状(2方向以上・延長250m以上)であることに加えて、
  `landuse=residential` が近くにあるか、実OSM建物(`hasRealBuildingNearby`。手続き生成でない本物)
  が近くに実在するかのどちらかを裏付けとして要求する。農地の農道グリッドを住宅街と誤認して
  家を乱立させる問題への対策。
- **学校・大学・病院の敷地(campus)**: Overpassで `amenity=school|university|college|hospital` の
  面(way)も取得し、敷地全体を回避ゾーン(`avoidPolygons`)にしている。敷地内に手続き生成の家は
  建たない。校舎そのものに`building`タグが無く敷地(amenity)しかマッピングされていない場合も、
  敷地中心に代表的な校舎を1棟生成する(サイズは敷地全体でなく34m×22mにキャップ)。
- **校庭**: 土の粒状テクスチャ+陸上トラックの白線(スタジアム型6レーン)を1枚のcanvasに焼き込んだ
  `campusGroundMat` を使用。span<900mまで描画。
- **地形追従の面ポリゴン(`buildTerrainFollowingAreaPoly`)**: 公園(lawn)・田畑(farm)・校庭(campus)
  の地面は、元のOSM way頂点(数個)だけを結ぶ平らな三角形(`buildAreaPoly`)だと起伏のある地形で
  浮いたり埋まったりする問題があったため、外接矩形をcellSize間隔の格子に分割してgetGroundYを
  細かくサンプルする方式に変更済み。水域(`waterAreaMat`)は意図的に据え置き(水面は本来平ら)。
- **駅舎(現実モード)**: 従来の「金色の塔+光る星+回転リング」というファンタジー演出をやめ、
  最寄りの線路脇に実際の駅舎(壁+屋上パラペット+ガラス張り改札+庇+時計)を生成する。駅名看板は
  駅舎の上空(地上+30m)に浮かせて遠くからでも見えるようにしてある。江戸/メルヘン/宇宙モードでは
  従来のファンタジー演出のまま(未変更)。
- **実OSM建物 vs 手続き生成建物の区別(`real`フラグ)**: `addBuilding()`の最終引数・
  `buildingRecords`の`rec.real`で管理。`hasRealBuildingNearby()`が空間検索に使う。

## 撤去済み・見送り中の機能

- **classifyCommercial(未分類の実建物をタグ不明でも周辺landuseから店舗判定)**: 実装→性能検証で
  重大な問題が発覚し撤去済み。**再実装する場合は必ず対策してから**(詳細は下記「学んだこと」参照)。
- **街路樹の実座標配置・信号機/ベンチ等の街路備品・一般道の簡易橋**: 実装したが動作不安定化の原因
  切り分けが難しく、ユーザー判断でロールバック。個別に再検証すれば復活の余地はあるが優先度低。
- **Claude APIを使った国・地域別の建物スタイル生成(ランタイム)**: アイデアとして出たが、
  レイテンシ・コスト・非決定性の理由で不採用。代わりに実タグ(landuse密度・building:levels等)で
  地域差を表現する方針に。
- **tile.openstreetmap.jp(TileServer GL / OpenMapTilesベクトルタイル)の活用**: 調査済み、不採用。
  landuseレイヤーは`class`一つのみでOSM生タグ以上の情報は無く、buildingレイヤーも既存のOverpass
  取得と同じ情報の焼き直しだった。乗り換えるメリットなしと判断。
- **Kanagawa地域のGeofabrik `.osm.pbf` 一括取り込み(Overpassのライブ依存を減らす)**: 提案のみ、
  「データ取り込みは後に回そう」とのことで未着手。
- **国土数値情報(KSJ)「土地利用細分メッシュ」の統合**: 提案のみ、未着手。OSMのlanduseタグより
  細かい・公式なJapan土地利用データとして、landuseリアリティを上げたくなったら検討候補。
- **Supabase/Upstash等の外部DBをOverpassキャッシュに使う案**: Renderの無料枠にディスクが無く
  `server/cache/`が再デプロイで揮発する問題への対策として議論。未実装。

## 学んだこと・ハマりどころ

- **`classifyCommercial`の教訓**: 実OSMのlanduseポリゴン(住宅地・商業地)は広範囲・高頂点数になり
  がち。`polyGridAdd`はポリゴンのbboxが触れる全グリッドセルに登録するため、未分類の建物1件ごとに
  近傍ポリゴンを検索して`pointInPolygon`するような処理を入れると、初回ロードが激重化し、メイン
  スレッドが詰まってフレーム駆動のチャンク生成・道路拡張まで止まって見える。**建物単位のループ内で
  landuseポリゴンを検索する処理は要注意**。エリア単位(道路のdenseArea判定のように、チャンク生成
  1回につき1回程度)に留めること。
- **面メッシュ(`buildAreaPoly`)は起伏地形で浮く/埋まる**: OSM wayの頂点は疎(数個)なため、
  そのままShapeGeometry化すると頂点間で平らな大きい三角形になり、地形の起伏を無視する。実際に
  プレイヤーが歩く地物(校庭・田畑・公園)は`buildTerrainFollowingAreaPoly`(格子分割+個別サンプル)
  を使うこと。水域のような「本来平らであるべき」地物は従来の`buildAreaPoly`のままでよい。
- **UV座標の注意**: `THREE.ShapeGeometry`のデフォルトUVは**ワールド座標そのもの**(0..1正規化では
  ない)。`farmMat`のようにrepeatラップ前提のテクスチャはこれでよいが、「ポリゴン1枚にきっちり
  収めたいテクスチャ」(校庭のトラック線など)には向かない。`buildTerrainFollowingAreaPoly`は
  `worldUV`引数でどちらのモードも選べるようにしてある。
- **git運用**: ユーザーはPowerShellで自分でgit操作する。Claudeはコマンドを提示するだけ。**複数機能を
  一度に実装した場合、`git add -A`で意図せず全部まとめてコミットされる**ので、機能ごとに個別コミット
  したい場合は都度指示すること(過去に混在コミットをやってしまった反省あり)。
- **このセッションのサンドボックス(bash)固有の注意**: `index.html`を`mcp__workspace__bash`経由で
  読むと、あるタイミングから**ファイルサイズが更新されず古いスナップショットに固定される**現象が
  複数回発生した(`wc -c`が同じ値のまま変わらず、末尾がUTF-8マルチバイト文字の途中で切れる)。
  Editツール自体は正常に反映されている(Read toolでは最新内容が見える)ので、bash側の`node --check`
  が失敗/古い内容を検証している場合は、Read toolで該当箇所を目視確認する方が信頼できる。
- **Renderの無料枠**: 永続ディスク非対応。`server/cache/`は再デプロイのたびに消える。Overpassの
  レート制限(1ホスト1.1秒間隔)は`server/server.js`でホスト単位に直列化しており、**端末に関係なく
  共有**(同じRenderインスタンスに複数端末が同時アクセスしても1つのキューを共有する)。

## 今回セッションの経緯(時系列サマリ)

1. 屋根・外壁の色/形をOSMタグから反映する機能を実装 → 4機能まとめて実装していたことに起因する
   コミット混在ミスが発覚 → 手動でロールバックし、屋根・外壁機能だけ残す。
2. 「高速道路上で見えないオブジェクトに動きをブロックされる」バグ報告 → 橋脚を視覚・物理とも撤去 →
   それでも残る階段状の当たり判定バグを3回の切り分けで根本解決(最終的に連続勾配式の
   `motorwaySlopes`に置き換え、Box3ベースの当たり判定を全廃)。
3. OSMグリッドフォールバック(道路が格子状になる現象)の原因調査 → Overpassレート制限+Renderの
   ディスク非永続化の組み合わせと判明。
4. データ取り込み(Geofabrik一括インポート)・Claude APIランタイム活用・OSM Japanタイルサーバー
   活用、といった大きな拡張案を検討 → いずれも「今回は見送り、実タグドリブンな方針を継続」に着地。
5. 建物高さ・マンション判定/住宅地タイトニングを再実装(以前ロールバックで巻き込まれた機能の復活)。
6. ユーザーから8項目の要望(学校・病院を際立たせる/学校敷地に家を建てない/商業施設を家と区別/
   住宅乱立防止/マンション・工場を大きく/農地に家を建てず畑や田んぼを/公園に家を建てない/
   立派な駅舎)→ 一括実装 → **動作が重くなったとの報告**。
7. 全体をロールバックし、**1機能ずつ実装→確認**する方式に切り替え:
   1. 駅舎の実装 → OK
   2. 中立色・サイズ底上げ・ランドマーク最低高さ・煙突 → OK
   3. denseAreaの実建物裏付け条件 → OK
   4. classifyCommercial(未分類建物の商業判定)→ **NG、激重化+タイル処理が詰まる** → ロールバック
   5. 学校・病院の敷地回避ゾーン → OK
8. 追加の磨き込み:
   - 校舎にbuildingタグが無い学校で校舎が生成されない問題を修正(敷地amenityのみでも校舎を1棟生成)
   - 校庭に土テクスチャ+陸上トラック線を追加
   - 校庭・田畑(・公園)が起伏地形で浮く/埋まる問題を、地形追従メッシュへの置き換えで修正
9. 本ログを作成(このファイル)。
