# 高層ビル生成問題 デバッグログ (2026-07-16)

「超高密度エリア(京橋・八重洲・新川・八丁堀等)で大きい商業ビルが建たず空き地になる」問題の完結編。
前史(Overpass信頼性のv1〜v7)は HANDOFF_20260716.md と過去ログ参照。

## 真因(2つの独立した問題が重なっていた)

### 1. ネットワーク層: Overpassの無言の部分応答(v4〜v7で対処)
公開Overpassインスタンスは負荷次第で、エラーもremarkも無しに「応答できた分だけ」を200 OKで返すことがある。
- 対策: クエリに `out count;` を追加し、宣言総数と実受信数を照合(part8.js)。不一致は失敗→再試行。
- server.js も同じ検証を行い、**不完全応答はディスクキャッシュしない**(以前は汚染キャッシュが永久にHITし続けた)。
- 直接モード(プロキシ不調時)はミラー輪番(overpass-api.de / kumi.systems / private.coffee)+ホスト別ペース配分+429/5xxバックオフに。
- `OSM_TILE_BATCH` は 1→3 に復帰(count検証があれば部分応答は検出できる。1タイル化はリクエスト数増で429ストームの主因だった)。

### 2. 処理層: isOnRoadの外接円判定が大型ビルを決定論的に破棄(本セッションの主発見)
part9.jsの生成ループが全建物に `isOnRoad()` をかけており、判定は外接円半径(対角線の半分)。
60m×40mのビルはhalfDiag≈36mなので、周囲に道路がある街区では必ず「道路上」と判定→黙って破棄。
さらにknownBuildingGrid登録済みのため手続き生成もブロック=完全な空き地。
**「大きいビルだけ消える」「消える場所が毎回同じ」の正体。**
- 対策: 実OSM建物(real)はisOnRoadチェック免除。

## 破棄をやめたことで露見した表現の問題と対策

1. **建物が全て軸平行・巨大化して道路に覆いかぶさる**
   → 旧実装は「重心+軸平行外接矩形」(45°回転の100×20mビルが85×85mになる)。
   フットプリント最長辺の方位角で回転座標系に投影し、実際の向き・寸法・中心を復元(part8.js)。
   メッシュは生成後に建物中心周りで剛体回転(part3.js addBuilding、rotパラメータ)。

2. **当たり判定と見た目のズレ**
   → cboxは回転フットプリントの外接AABB(グリッド登録用)+回転情報(rot/cx/cz/hw/hd)。
   wouldCollide/floorHeightAtはプレイヤー座標を建物ローカル系へ逆回転して判定(part7.js collBoxHitsXZ)。

3. **道路・線路リボンへの食い込み**(ゲーム側リボンは実際より広いため)
   → fitRealBuildingToRoads(part2.js): 建物ローカル系で近傍セグメントを評価し、
   中心固定で幅・奥行きを道路縁+0.5mまで縮小。下限は元の35%。水面リボンは除外。

4. **線路は一切被り禁止**(駅ビルの半透明・透過コリドー案は試作後に撤去)
   → 線路も縮小対象+縮小後の最終チェックでまだ被る建物(線路またぎ)は生成スキップ(drop)。

5. **ミニマップとの食い違い** → minimapBuildingsにrotを持たせ、canvas回転で描画(part7.js)。

6. **ビル色が単調** → 現実モードで6種の量子化ティント(明暗・寒暖・ガラス風)を壁色に適用(part3.js tintWall)。
   量子化なのでマテリアルキャッシュは増殖しない。神社仏閣・教会は除外。

## 教訓
- 症状が場所に対して完全に安定して再現するなら、確率的なネットワーク説より決定論的なデータ依存処理を疑う。
- 「成功扱いなのにデータが無い」系は、取得層だけでなく**受信後に黙って捨てる経路**を全部洗う。
- Chromeコンソールは失敗リクエストしか表示しない(ミラー輪番が「効いていない」ように見えた罠)。

# 後続対応 (同日・高層ビル一区切り後)

## 水面1m下降補正 → リバート(教訓)
「道路が川・海に埋まる」対策として水面ポリゴン+0.15→-0.85、リボン+0.05→-0.95を試したが、
見た目が崩れて即リバート。**水面のyOffは +0.15(ポリゴン)/+0.05(リボン) が正**。
道路水没問題は未解決のまま保留(再挑戦時は一律下降ではなく、道路との重なり箇所だけの局所処理を検討)。

## ジャンプ後ロード時間の改善
- 遠距離ジャンプは location.reload() 経由のフル再起動(jumpToLatLon、part7.js)なので、
  改善対象は「リロード後の初期ロード全体」。
- OSMタイル初期先読み 7×7(49枚)→5×5(25枚)。5×5でも最低±3200mで
  ROAD_UNLOAD_DIST(2500m)・実建物生成(3000m)をカバー。
- OSM_TILE_CONCURRENCY 2→3(直接モードはミラー輪番3ホストなので安全)。
- 初期ラッシュ: 起動後30秒は建物生成予算を400棟/フレーム・14ms上限に拡大
  (通常時160棟・8ms。道路混雑時の絞りも5→40)。30秒後は自動で通常に戻る。
- ジャンプ用マップの「現在地」吹き出し(bindPopup)を削除(🧙マーカーのみに)。

## 建物の生成・消去距離を種類別に分離
- 実OSM建物: BUILDING_GEN_DIST_REAL=3000m / BUILDING_UNLOAD_DIST_REAL=3800m
  (unloadFarBuildings・reactivateNearbyDormantBuildings・part9生成ゲートをreal別分岐に)。
- 手続き生成: CHUNK_RADIUS 3→8(960m≒1000m、消去は+2=1200m)。
- 森(FOREST_R)はCHUNK_RADIUS連動をやめて480m固定(樹木負荷の暴騰防止)。
- FPSが厳しい場合の調整候補: 実建物2000〜2500m、CHUNK_RADIUS 6(720m)。
