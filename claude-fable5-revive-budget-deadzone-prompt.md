# 相談プロンプト: REVIVE_BUDGETが「records=bMaxの95%ちょうど」で完全ゼロになるデッドゾーンを実機で確認

以下をそのままFable 5に貼ってください。文脈: `IMPL_PROMPT_20260721_DORMANT_REVIVAL.md`で
あなたに確認してもらった「空間グリッド化+容量ベースREVIVE_BUDGET+表示分離」は既に実装済みで、
実機ログでも良好に機能していた(dormant 51000→23000台まで減少、revived/2sが上限600で
安定推移)。この時点であなたは現行設計を妥当と判断し、bMax動的引き上げ・進行方向優先機構は
「不要」と結論していた。しかしその直後、別の実機ログで具体的な反例が取れたので再検証したい。

## 現行実装(part1.js、変更なし)

```js
function reactivateNearbyDormantBuildings() {
  _dormantCheckFrame++;
  if (_dormantCheckFrame % 90 !== 0) return;
  if (dormantCount === 0) return;
  const REVIVE_BUDGET = Math.max(0, Math.min(600, Math.floor(PERF.bMax * 0.95 - buildingRecords.length)));
  if (REVIVE_BUDGET === 0) return; // ここで完全停止する
  ...
}
```

## 実機で観測した反例

同じセッション内、直前直後の2回のログ抜粋:

```
(直前) records 11594/12000(96.6%) dormant 23322  revived/2s 600  generated/2s 4858  toDormant/2s 1383
(今回) records 11400/12000(95.0%) dormant 75932  revived/2s 8    generated/2s 112   toDormant/2s 2674
```

プレイヤーが移動し、`toDormant/2s`(evict)が2674と高水準を維持したままdormantが
23322→75932へ急増(+52610)している。この間 records は12000*0.95=11400ちょうどに
張り付いており、`REVIVE_BUDGET = floor(11400 - 11400) = 0`。式の定義上、recordsが
95%ラインに到達・張り付いた瞬間に空き枠が文字通りゼロになり、`if (REVIVE_BUDGET === 0) return;`
で復帰処理そのものが丸ごとスキップされる。

## 見立てている因果(検証してほしい)

1. 「95%までの空き枠」という設計は、recordsがそのラインより十分下にある間は機能するが、
   ライン**ちょうど**、あるいはそれ以上(evictで一時的に押し戻される場合を含む)では
   構造的に0になる。これは密集地でプレイヤーが動き続けている間、evict(toDormant)と
   revive(reactivate)が同時に激しく起きるべき状況で、revive側だけが完全停止するという
   非対称を生んでいる。
2. 前回の相談時点(revived/2s 600で安定)では records が94%台など95%未満で推移していた
   ため、この境界ケースが実機データに現れていなかった可能性が高い。今回たまたま
   ちょうど95%に張り付く場面を捕捉できた。
3. 完全ゼロではなく、常時ごく少量(例: 20〜30件/サイクル)を流す下限を設ければ、
   雪崩スパイク(あなたが元々REVIVE_BUDGET導入で防ごうとしていたもの)を再発させずに
   このデッドゾーンだけ塞げるのではないか。

## 相談したいこと

1. 上記の見立ては妥当か。「95%ちょうど、またはそれ以上」で完全ゼロになる設計は
   意図した挙動か、見落とされていた境界ケースか。
2. 対策として「`Math.max(小さな下限, ...)`で完全ゼロを避ける」という修正で十分か、
   それとも別の設計(例: 95%という閾値自体を下げる、evictと同じサイクルでrevive枠を
   確保する、等)の方が良いか。
3. 下限を設ける場合、具体的にどの程度の値が安全か(記録上限12000に対して20〜30件/
   サイクルという私の初期案は妥当か、それとも別の基準で決めるべきか)。

コードの実装自体は別チャットで行う前提。まずは原因の妥当性と対策の方針だけ検証してほしい。
