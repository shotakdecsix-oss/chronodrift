# 相談プロンプト: 密集地で建物生成が「取得後」の段階で詰まる問題

以下をそのままFable 5に貼ってください。文脈: 別チャットでOverpass API側(429/504対策、
OSM_TILE_CONCURRENCY調整、bMax飽和時のdormantBuildings無限増加=一方通行の片道弁)を
一通り修正・実機確認済み(dormantは74000〜85000から34000台まで低下、正しく機能している)。
その後、東京駅周辺で「建物生成が黄色(buildingPending)から進まない」という報告が続き、
データ取得側と生成側のどちらが真因か切り分けるための計測を入れたところ、生成側が
支配的というデータが取れた。この解釈と対策方針を検証してほしい。

## 実機データ(2026-07-21、東京駅周辺、std設定)

```
[fetch] active 3 queue 18 cooldown(ms) 0 streak 0 records 7119 / 12000 dormant 64696
[buildgen] budget 40 roadBacklogGate 552 rush true generated/2s 672 requeued/2s 4522 toDormant/2s 2834 pendingTotal 66708
```

`records`はbMax(12000)にまだ余裕があり、`cooldown`も0(Overpass側は今は健全)。それにも
関わらず、直近2秒間で実際に生成まで進んだ建物はわずか672件、それに対しチャンク地形/
周辺タイル未確定で列の末尾に回された(空回り)件数が4522件、dormantへ退避した件数が
2834件。処理試行の内訳では**生成成功はわずか8.4%(672/8028)、56.3%が空回りの再キュー**
という比率になっている。ユーザーからは「タイルに突入する直前まで地形が青(waitTerrain)の
まま、地形が緑になっても道路が黄色/赤のままのケースが見える」という目視報告もある。

## 関連コード(line番号は`js/legacy/`基準、内容は実コード確認済み)

- `part9.js`(exploreOnUpdate内、建物生成ループ): 1フレームあたりの生成予算`_buildBudget`
  (バックログに応じ最大160、rush時最大400)。ただし`pendingRoadMeshes.length`(道路メッシュの
  グローバル残件数)が80を超えると、rush中でも40、非rush時は5まで強制的に絞られる。
  今回のログでは`roadBacklogGate=552`(閾値の7倍)なので常時絞られている状態。
- 同ループ内: 各建物候補は`chunkNearTerrainReady(bcx,bcz)`(地形チャンク単位の準備確認)と
  `osmTilesReadyAround(b.x,b.z,64)`(周囲64m以内の全OSMタイルの道路確定待ち)の両方を通らないと
  生成されない。どちらか未達なら`b._tries`をインクリメントし、40回未満なら
  `pendingBuildings.push(b)`(配列の**末尾**へ)して`continue`(=このフレームの予算を1件
  消費するが生成はしない)。これが今回計測した「requeued」。
- `part6.js`地形(NEAR)取得:
  - `NEAR_HALF_LAT = NEAR_HALF_LON = 0.036`(≈±4km四方)、`checkNearTerrain`の再取得トリガーは
    `player位置がnearCXから NEAR_W*0.4(≈3200m)離れたら`。窓の半幅4000mに対し残り800mの
    バッファしかなく、ダッシュ最大45m/sなら約18秒で窓の端に到達する計算。
  - **重要**: NEAR取得(`loadNearTerrain`)は441点(21×21グリッド)を**まとめて1回**で取得する
    設計で、部分更新は無い。取得完了まで`nearElev/nearCX/nearCZ`は古い(前回の)ままなので、
    取得完了までの間は新しいエリアの`terrainReady`判定がすべて古い窓基準で行われ続ける。
  - `fetchElevationsGSI`(part5.js:196-223)は、441点をタイル単位に重複排除し
    (z14で概算16〜25枚程度)、`runLimited(..., 8)`で8並列取得する。個別タイルの404は
    「データ無し(海上等)」として正常に`null`を返す設計(part5.js:168、コメント上も
    「対応不要」と確認済み)。**しかし404以外のエラー(5xx・ネットワーク例外)が1枚でも
    起きると、try/catchでfetchElevationsGSI全体がnullを返し、441点まるごと
    opentopodata(`part6.js:216-227`、1リクエスト/秒・1回100点=5バッチの逐次実行という
    重い制限)へフォールバックする**。この全体フォールバックが発生すると、通常「数秒」の
    NEAR取得が数十秒規模まで悪化しうる。実機コンソールでは今回GSIタイルの404が
    複数連続して観測されている(404自体は想定内動作のはずだが、同時に他の理由で
    フォールバックが発生していないかは未確認)。
  - `onNearTerrainFail`はopentopodata側の失敗時のみ呼ばれ、失敗するたび
    `checkNearTerrain`の次回チェック間隔を30フレーム→最大600フレーム(10秒)まで
    伸ばす(`_nearFailCount`)。GSI成功時はこのカウンタに触れない。

## 見立てている因果(検証してほしい)

1. 密集地(東京駅)ではOSMタイル取得(3並列、タイルごとに独立進行)の方が、NEAR地形の
   「全窓一括・単発」取得より粒度が細かく並列度も高いため、**建物データの到着ペースに
   地形の網羅ペースが追いつかない**瞬間が生まれる。
2. 追いつかない間、`chunkNearTerrainReady`に落ちた建物候補は生成されず、`_tries<40`の間
   ひたすら配列末尾への再挿入を繰り返す。これが生成予算の56%を空費し、実効スループット
   (8.4%)を大きく引き下げている。
3. さらに`roadBacklogGate`(道路メッシュ残件数80超で予算を5〜40に強制圧縮)が、
   密集地では常時ほぼ張り付いており、ただでさえ低い実効スループットの絶対量自体も
   絞っている。
4. 加えて、GSI取得中に404以外の何らかのエラーが1件でも起きると441点まるごと
   opentopodataへ全滅フォールバックする設計は、密集地で地形取得が「たまに」大きく
   遅延する不安定要因になっていそうだが、今回のログだけでは実際にこの全滅フォールバックが
   発生したかどうかまでは確認できていない。

## 相談したいこと

1. 上記1〜4の因果の見立ては妥当か。見落としている経路(例: chunkNearTerrainReadyが
   実は別の理由で頻繁に不成立になっている等)はあるか。
2. 対策の優先順位はどこに置くべきか。候補: (a) NEAR取得のトリガー閾値・窓サイズを
   見直し、密集地移動時にもっと早く/こまめに追従させる、(b) `_tries`超過前に配列末尾への
   即時再投入をやめ、地形未準備の建物を別のペンディング列(dormant的な待避列)へ逃がして
   本流の予算を空回りさせない、(c) `fetchElevationsGSI`の部分エラーを「タイル単位の
   個別リトライ」にし、1枚の失敗で441点全体をopentopodataへ道連れにしない、(d) その他。
3. 過去にGPUクラッシュ(建物+道路メッシュ総量が主因、bMaxキャップ等で対処済み)の
   経緯があるため、生成予算(_buildBudget)や関連キャップを緩める方向の対策は慎重に
   検討したい。緩めずに(a)〜(c)のような「無駄な空回りを減らす」アプローチだけで
   十分な改善が見込めるか、それとも予算側の見直しも必要か。
4. 実装順として、まず(c)のGSI部分エラー耐性(影響範囲が小さく、他システムとの
   相互作用も少ない)から着手するのが妥当に見えるが、優先順位について助言がほしい。

コードの実装自体は別チャットで行う前提。まずは原因の妥当性と対策の方針だけ検証してほしい。
